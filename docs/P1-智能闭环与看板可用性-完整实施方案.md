# P1（2~4周）智能闭环与看板可用性实施方案

> 适配你当前项目包结构：`com.ityfz.yulu`。
> 目标：
> 1) 补齐意图识别落库（`chat_message.intent`）
> 2) 情绪+意图标签进入客服工作台（列表+实时推送）
> 3) 看板新增：意图分布、负向情绪率、低分告警
> 4) 满意度链路闭环（提交 -> 处理 -> 追踪）

---

## 0. 基线说明

你当前代码已经具备：
- 会话消息表：`chat_message`（含 `emotion`、`intent` 字段）
- 转人工服务：`HandoffService`
- 评分服务：`HandoffRatingService` + `AdminHandoffRatingController` + `CustomerHandoffRatingController`
- 看板入口：`AdminDashboardController#overview` + `AdminDashboardService`

本文给的是**可直接落地的 P1 方案**，优先保证：
- 后端查询口径统一（租户隔离）
- 实时链路不中断（WebSocket失败可降级）
- 前端工作台与看板可直接消费

## 1) 意图识别落库（chat_message.intent）

## 1.1 目标行为

每次用户提问：
- 识别情绪 `emotion`
- 识别意图 `intent`
- 以用户消息维度落库到 `chat_message`

AI消息不承载用户情绪，建议固定：
- `AI.emotion = NEUTRAL`
- `AI.intent = null`

---

## 1.2 SQL（若已有字段可跳过）

```sql
ALTER TABLE chat_message
  ADD COLUMN intent VARCHAR(32) NULL COMMENT '意图: GENERAL/REFUND/INVOICE/LOGISTICS/COMPLAIN',
  ADD INDEX idx_cm_tenant_session_sender_time (tenant_id, session_id, sender_type, create_time),
  ADD INDEX idx_cm_tenant_intent_time (tenant_id, intent, create_time),
  ADD INDEX idx_cm_tenant_emotion_time (tenant_id, emotion, create_time);
```

## 1.4 LLMClient 输出规范

文件：`src/main/java/com/ityfz/yulu/common/ai/impl/LangChain4jQwenClient.java`

保证 `detectIntent` 只输出规范值：
- `GENERAL/REFUND/INVOICE/LOGISTICS/COMPLAIN`

在 `detectIntent` 末尾统一：

```java
private String normalizeIntent(String intent) {
    if (intent == null) return "GENERAL";
    switch (intent.toUpperCase()) {
        case "REFUND":
        case "INVOICE":
        case "LOGISTICS":
        case "COMPLAIN":
        case "GENERAL":
            return intent.toUpperCase();
        default:
            return "GENERAL";
    }
}
```

新增私有方法：

```java
private ChatMessage queryLatestUserMessage(Long tenantId, Long sessionId) {
    return chatMessageMapper.selectOne(
            Wrappers.<ChatMessage>lambdaQuery()
                    .eq(ChatMessage::getTenantId, tenantId)
                    .eq(ChatMessage::getSessionId, sessionId)
                    .eq(ChatMessage::getSenderType, "USER")
                    .orderByDesc(ChatMessage::getCreateTime)
                    .last("LIMIT 1")
    );
}

private String calcRiskLevel(String emotion, String intent) {
    String e = emotion == null ? "NEUTRAL" : emotion.toUpperCase();
    String i = intent == null ? "GENERAL" : intent.toUpperCase();

    if (("ANGRY".equals(e) || "NEGATIVE".equals(e)) && "COMPLAIN".equals(i)) {
        return "HIGH";
    }
    if ("ANGRY".equals(e) || "NEGATIVE".equals(e) || "COMPLAIN".equals(i)) {
        return "MEDIUM";
    }
    return "LOW";
}
```

## 3.2 Mapper SQL

文件：`src/main/java/com/ityfz/yulu/admin/mapper/AdminDashboardMapper.java`

```java
@Mapper
public interface AdminDashboardMapper {

    // 已有：querySessionTrend/queryHandoffTrend ...

    @Select("""
        SELECT IFNULL(intent, 'GENERAL') AS intent, COUNT(1) AS cnt
        FROM chat_message
        WHERE tenant_id = #{tenantId}
          AND sender_type = 'USER'
          AND create_time >= #{fromTime}
        GROUP BY IFNULL(intent, 'GENERAL')
        ORDER BY cnt DESC
        """)
    List<DashboardIntentPointVO> queryIntentDist(@Param("tenantId") Long tenantId,
                                                 @Param("fromTime") LocalDateTime fromTime);

    @Select("""
        SELECT
            IFNULL(SUM(CASE WHEN UPPER(IFNULL(emotion,'NEUTRAL')) IN ('ANGRY','NEGATIVE') THEN 1 ELSE 0 END), 0) AS negCnt,
            COUNT(1) AS totalCnt
        FROM chat_message
        WHERE tenant_id = #{tenantId}
          AND sender_type = 'USER'
          AND create_time >= #{fromTime}
        """)
    Map<String, Object> queryNegativeEmotionRate(@Param("tenantId") Long tenantId,
                                                 @Param("fromTime") LocalDateTime fromTime);

    @Select("""
        SELECT
            id AS ratingId,
            handoff_request_id AS handoffRequestId,
            session_id AS sessionId,
            user_id AS userId,
            agent_id AS agentId,
            score,
            comment,
            submit_time AS submitTime
        FROM handoff_rating
        WHERE tenant_id = #{tenantId}
          AND submit_time >= #{fromTime}
          AND score <= #{maxScore}
        ORDER BY submit_time DESC
        LIMIT #{limit}
        """)
    List<DashboardLowScoreAlertVO> queryLowScoreAlerts(@Param("tenantId") Long tenantId,
                                                       @Param("fromTime") LocalDateTime fromTime,
                                                       @Param("maxScore") Integer maxScore,
                                                       @Param("limit") Integer limit);
}
```

## 4) 满意度链路闭环（提交 -> 处理 -> 追踪）

你现有 `handoff_rating` 已基本满足，建议补一张处理轨迹表，提升可审计性。

## 4.1 SQL

```sql
CREATE TABLE IF NOT EXISTS handoff_rating_track (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id BIGINT NOT NULL,
  rating_id BIGINT NOT NULL,
  action VARCHAR(32) NOT NULL COMMENT 'SUBMIT/PROCESS/CALLBACK_DONE/REOPEN',
  operator_id BIGINT NULL,
  note VARCHAR(500) NULL,
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_tenant_rating_time (tenant_id, rating_id, create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

---

## 4.2 处理流程状态机

`handoff_rating.status`：
- `WAITING`：待评价
- `RATED`：用户已提交
- `PROCESSED`：管理员已处理
- `EXPIRED`：超时未评价

可扩展：
- `CALLBACK_DONE`：回访完成（可作为 `PROCESSED` 的细分）

---

## 4.3 Service 关键改造

文件：`src/main/java/com/ityfz/yulu/handoff/service/Impl/HandoffRatingServiceImpl.java`

在三个节点写轨迹：
- `submit(...)` -> insert track: `SUBMIT`
- `process(...)` -> insert track: `PROCESS`
- 新增 `callbackDone(...)` -> insert track: `CALLBACK_DONE`

示例（简化）：

```java
private void addTrack(Long tenantId, Long ratingId, String action, Long operatorId, String note) {
    HandoffRatingTrack t = new HandoffRatingTrack();
    t.setTenantId(tenantId);
    t.setRatingId(ratingId);
    t.setAction(action);
    t.setOperatorId(operatorId);
    t.setNote(note);
    trackMapper.insert(t);
}
```

## 5) 联调步骤（建议顺序）

1. **后端落库改造**：`ChatServiceImpl`（emotion/intent）
2. **客服工作台标签**：`HandoffService + AgentWorkbenchPage`
3. **看板扩展**：`Mapper SQL + VO + Service` 聚合
4. **满意度闭环**：`track` 表 + `Service` 埋点 + 查询接口
5. **前端看板消费**：
   - KPI 卡新增：负向情绪率、平均满意度
   - 图表新增：意图分布（饼图/条形）
   - 告警卡新增：低分评价 TopN

## 6) 验收清单

- [ ] 用户发消息后，`chat_message.intent` 非空（至少 GENERAL）
- [ ] 客服待处理列表可见 `emotion/intent/risk`
- [ ] 实时新请求无需刷新可见标签
- [ ] `/api/admin/dashboard/overview` 返回 `intentDist`、`negativeEmotionRate`、`lowScoreAlerts`
- [ ] 评分从提交到处理可追踪（含操作人、时间、备注）

---

## 7) 风险与兜底

- LLM 输出非 JSON：必须走规则回退（`fallbackRuleIntent/fallbackRuleEmotion`）
- WebSocket 推送失败：不影响主流程，前端可轮询补偿
- 低分告警阈值可配置：建议 `score <= 2`
- 强约束租户隔离：所有 SQL 必带 `tenant_id`

---

## 8) 你可以直接先落地的最小提交（建议）

1. `ChatServiceImpl`：意图/情绪落库（1次提交）
2. `HandoffService`：工作台标签 + 实时 payload（1次提交）
3. `AdminDashboardMapper/Service/VO`：看板三项增强（1次提交）
4. `HandoffRatingTrack` + 查询接口（1次提交）

这样拆分最稳，便于回滚。
