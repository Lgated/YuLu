AI 智能客服中台 ≠ 单点“对话机器人”，而是**公司级能力平台**：  
一边把各渠道（热线、APP、小程序、抖音、钉钉…）的咨询接进来，  
另一边把知识、算法、数据、人力、工单、营销能力沉淀下来，**让任何业务线 1 天就能长出客服机器人**。

下面按“业务 → 技术”两层，把**真正决定生死的模块**一次性列给你，并给出落地形态（含开源/商业选型），可直接抄作业。

## 一、业务层（7 大核心）

表格

复制

| 模块                    | 不做会死的地方                    | 落地形态（可抄）                                                                          |
| :-------------------- | :------------------------- | :-------------------------------------------------------------------------------- |
| 1. 全渠道消息总线            | 每新增一个渠道就要重新开发接入、维护 10 套长连接 | 自研网关 + 统一事件模型；WebSocket/电话线路用 Asterisk/FreeSWITCH；微信/抖音用官方云                       |
| 2. 知识中枢（KG + QA 对）    | 没有知识图谱，机器人永远“搜关键词”，答非所问    | Neo4j 存实体关系 + Elasticsearch 存 QA 对；运营后台支持“可视化节点拖拽”                                |
| 3. 运营工作台（标注→训练→评测→上线） | 没有闭环，运营 1 周就放弃机器人          | 标注用 Label-Studio；训练用 Jupyter + MLflow；评测用 4 级指标（召回、精确、F1、满意度）                     |
| 4. 人机协同（坐席辅助 + 无感转人工） | 机器人解决率 60% 天花板，转人工体验差导致投诉  | 坐席端用 Electron 套壳；转人工带上下文（消息追溯 ID）+ 路由策略（熟客优先 / 技能组 / 空闲度）                         |
| 5. 工单 & 质检一体化         | 对话结束就丢失，无法追溯、无法考核          | 对话自动生成工单（OpenAPI 对接内部 OA）；质检用 ASR→文本→规则引擎（敏感词、SOP 时长）                             |
| 6. 数据看板（实时 + 离线）      | 老板问“机器人到底省了多少钱”答不上来        | 实时：Flink + Redis + Grafana；离线：Hive + Superset；核心指标：拦截率、转人工率、坐席饱和度、Cost-Per-Ticket |
| 7. 插件市场（让业务自助扩展）      | 每来一个新场景就排期开发，中台变瓶颈         | 插件=Docker 镜像；注册到 K8s CRD；中台只暴露 Event & API 规范，业务自己写插件（查物流、改地址、营销发券…）              |

## 二、技术层（8 大底座）

表格

复制

| 模块             | 关键设计点               | 选型/开源组合                                                                      |
| :------------- | :------------------ | :--------------------------------------------------------------------------- |
| 1. 事件驱动总线      | 消息必达、顺序、可回放         | Kafka + Kafka-Connect；每会话一个 partition → 保证消息顺序                               |
| 2. 对话引擎（DM）    | 支持多轮填槽、打断、上下文继承     | Rasa-Core / 自研有限状态机；槽位与业务实体字段绑定                                              |
| 3. NLP 工厂      | 意图识别、实体抽取、情感、FAQ 匹配 | 轻量：FastText+TextCNN；高精度：BERT+Sentence-Transformer；All-in-One：LangChain + LLM |
| 4. 语音链路        | 电话渠道必须支持 ASR+TTS    | ASR：Kaldi/FunASR/阿里云；TTS：FastSpeech2 + HiFi-GAN；电话网关：Asterisk + SIP trunk    |
| 5. 知识图谱服务      | 实体链接、属性查询、可视化       | Neo4j + nebula-graph + elasticsearch；提供 GraphQL 网关                           |
| 6. 弹性伸缩 & 灰度   | 白天高峰、晚上低峰，成本要可弹     | K8s HPA（CPU+队列堆积度）；蓝绿+金丝雀发布机器人模型                                             |
| 7. 实时特征存储      | 上下文、用户画像、坐席状态要毫秒级   | Redis + RedisSearch；向量缓存用 Milvus                                             |
| 8. DevOps 与可观测 | 机器人“说错话”要快速定位       | Log→Loki；Trace→Jaeger；Metric→Prometheus+Grafana；对话全链路 ID 透传                  |

## 四、 MVP 清单（30 天可上线）

1. 渠道：先接 Web 端（WebSocket）→ 后端统一事件模型
    
2. 机器人：FAQ 检索 + 关键字意图 → 拦截 40% 咨询
    
3. 知识：ES 存储 QA 对 + 运营后台增删改
    
4. 人机：机器人答不出 → 带上下文转人工（Web 坐席）
    
5. 数据：Kafka + MySQL + Grafana 看板（拦截率、转人工率）
    
6. 灰度：K8s 部署，按用户百分比放量

## 五、一句话总结

**业务上**先搞定“全渠道 + 知识运营 + 人机协同”三件套，  
**技术上**用“Kafka 事件总线 + 可插拔 NLP + K8s 弹性”做底座，  
**指标上**只盯“拦截率、转人工率、坐席成本”三个数，  
就能在 1 个月内拿出一个“公司级”智能客服中台 MVP，后续再逐步叠加语音、图谱、插件市场。
