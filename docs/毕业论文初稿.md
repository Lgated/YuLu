## 摘要

随着大语言模型与检索增强生成（Retrieval-Augmented Generation, RAG）技术的快速发展，智能客服系统正从“规则问答”向“知识增强的多轮对话平台”演进。传统客服系统在知识更新效率、上下文理解能力、人工介入协同以及多租户隔离方面存在明显不足，难以满足企业级 SaaS 场景下的稳定运营需求。本文面向企业服务场景，设计并实现了一个多租户 AI 智能客服中台系统。系统以 Spring Boot 与 React 为基础，融合 LangChain4j、Qdrant 向量检索、Redis 上下文缓存、RabbitMQ 异步事件以及 WebSocket 实时通信能力，构建了“AI 对话—情绪识别—转人工—工单处理”的完整业务闭环。
本文重点研究了四个核心问题：第一，多租户架构下的数据与检索隔离；第二，RAG 检索增强机制在客服问答场景中的落地；第三，转人工会话在排队、分配、接入、结束等环节的状态机设计；第四，基于消息事件驱动的工单协同流程。系统通过租户上下文注入与权限控制实现了租户级安全边界，通过向量检索与动态阈值策略提升了知识问答准确率，通过 Redis 客服状态心跳与会话负载控制保障了转人工稳定性，并通过可扩展的管理员视图支撑运营监控。
实验与验证结果表明：系统能够在多角色协同场景下稳定运行，具备较好的可维护性和扩展性；在知识密集型问答中，RAG 增强显著改善了回答相关性；在服务异常场景下，系统可通过工单兜底机制保证服务连续性。本文的工作为多租户智能客服中台建设提供了可复用的工程实践路径。

## Abstract

With the rapid development of large language models and Retrieval-Augmented Generation (RAG), intelligent customer service systems are evolving from rule-based QA to knowledge-enhanced multi-turn dialogue platforms. Traditional systems still suffer from low knowledge update efficiency, weak contextual understanding, poor human-agent collaboration, and limited tenant isolation, which are insufficient for enterprise SaaS operations.

This thesis designs and implements a multi-tenant AI customer service platform for enterprise scenarios. The system is built with Spring Boot and React, and integrates LangChain4j, Qdrant vector retrieval, Redis context cache, RabbitMQ asynchronous events, and WebSocket real-time communication. It forms a closed-loop workflow of “AI dialogue – sentiment detection – handoff to human agent – ticket processing”.

This work focuses on four key aspects: (1) data and retrieval isolation in multi-tenant architecture; (2) practical RAG enhancement for customer service QA; (3) state-machine-based handoff lifecycle including queueing, assignment, acceptance and completion; and (4) event-driven ticket collaboration.

The implementation demonstrates stable operation in multi-role collaboration scenarios and strong maintainability/extensibility. In knowledge-intensive QA tasks, RAG significantly improves relevance and reliability. Under service exceptions, fallback ticket mechanisms ensure continuity. This thesis provides a reusable engineering reference for building enterprise-level multi-tenant intelligent customer service platforms.

**Keywords**: Intelligent Customer Service; Multi-tenant; RAG; Human Handoff; Ticketing; WebSocket

## 1 绪论

### 1.1 研究背景

在数字化服务场景中，企业客服系统承担着售前咨询、售后支持、知识解答与满意度管理等关键任务。随着客户问题复杂度增加，传统 FAQ 与规则机器人难以满足动态、开放式问答需求。大语言模型（LLM）在自然语言理解和生成方面展现出显著优势，但纯模型回答容易产生“幻觉”，缺乏企业私域知识约束，难以直接作为生产级客服系统核心能力。

与此同时，SaaS 化企业服务对系统提出了更高要求：其一，不同租户之间必须实现严格数据隔离；其二，客服业务需要支持 AI 与人工协同；其三，系统需具备可观测、可管控、可扩展的中台能力。基于上述挑战，构建一个融合 RAG、多租户隔离、转人工与工单闭环的智能客服中台具有现实意义。

### 1.2 研究目的与意义

本文研究目标是设计并实现一个支持多租户、可扩展、可落地的 AI 智能客服中台，重点解决以下问题：

1. 在多租户场景下实现会话、知识库、工单、向量检索的隔离与安全访问；
2. 在客服问答中落地 RAG，提升回答准确性与可解释性；
3. 设计转人工状态机与实时通信机制，保障 AI 与人工服务平滑切换；
4. 构建情绪触发与工单兜底机制，提高异常场景下的服务连续性。

理论意义在于为“LLM + 多租户 SaaS + 客服业务流程”提供系统化实现路径；工程意义在于形成一套可复用的中台建设方案。

### 1.3 研究内容

本文围绕“系统架构—核心模块—流程闭环—测试验证”展开：

- 多租户架构与权限模型设计；
- AI 对话与上下文管理机制实现；
- RAG 检索增强流程与向量库集成；
- 转人工会话管理、客服状态管理与 WebSocket 实时通信；
- 工单事件流与异步处理机制；
- 系统功能与可用性验证。

## 2 相关技术分析

### 2.1 大语言模型与 LangChain4j

系统采用大语言模型作为自然语言生成与理解核心，通过 LangChain4j 统一封装模型调用、提示词构建与上下文拼装。该方式使业务层与模型供应商解耦，便于后续替换或扩展模型能力。

### 2.2 RAG 检索增强生成

RAG 将“检索”与“生成”结合：先从企业知识库检索相关片段，再将片段作为上下文注入提示词，引导模型回答。相比纯生成，RAG 更适合企业客服场景，能够降低幻觉概率并提供引用信息。

### 2.3 向量数据库 Qdrant

系统使用 Qdrant 存储文档向量与元数据，支持按租户过滤、TopK 相似度检索等能力。文档经分块与 embedding 后入库，实现私域知识可检索化。

### 2.4 Redis 与实时状态管理

Redis 在系统中承担两类角色：

1. 对话上下文缓存（提高多轮对话性能）；
2. 客服在线状态与会话负载管理（支持心跳 TTL、在线集合、当前会话数控制）。

### 2.5 RabbitMQ 异步事件机制

工单创建与通知处理采用异步事件机制，减少同步链路阻塞，提升高并发场景下系统吞吐与解耦性。

### 2.6 WebSocket 实时通信

WebSocket 用于用户与客服实时消息、转人工状态变更推送、系统通知推送。其低延迟特性满足人工会话与状态感知需求。

## 第三章 需求分析

### 3.1 角色与业务边界

系统主要角色包括：

- **用户（C端）**：发起会话、提问、申请转人工、查看状态；
- **客服（Agent端）**：接受请求、处理人工会话、结束会话；
- **管理员（Admin端）**：管理工单、监控客服状态、运营配置。

### 3.2 核心业务需求

1. 支持 AI 多轮对话并可结合知识库回答；
2. 支持转人工排队、分配、接入、结束全流程；
3. 无客服在线时可自动兜底创建工单；
4. 多租户下数据隔离、权限隔离、检索隔离；
5. 支持客服在线状态管理与会话负载控制。

### 3.3 非功能需求

- 可靠性：关键状态需落库，实时消息作为加速层；
- 安全性：租户隔离与角色权限控制；
- 可扩展性：模块化设计，支持后续功能增量；
- 可维护性：事件审计、文档化与分层结构。

## 第四章 系统总体设计

### 4.1 总体架构

系统采用前后端分离 + 中台化架构：

- 前端：React（C端、客服端、管理端）；
- 后端：Spring Boot（REST API + WebSocket + 事件处理）；
- 存储：MySQL（业务数据）、Redis（缓存与状态）、Qdrant（向量）；
- 消息：RabbitMQ（异步事件）；
- 调度：XXL-Job（可选任务调度）。

### 4.2 分层设计

- Controller：请求入口、参数校验、权限注解；
- Service：业务编排、状态流转、事件写入；
- Mapper：数据访问；
- WebSocket Handler：实时消息通道。

### 4.3 多租户隔离策略

通过租户上下文（tenantId）贯穿 Controller -> Service -> Mapper，实现：

- 业务数据按 tenantId 约束；
- 向量检索按 tenantId 过滤；
- 客服在线状态按 tenantId 维护独立 Redis key 空间。

### 4.4 核心状态机设计

以转人工为例，核心状态包括：`PENDING -> ASSIGNED -> ACCEPTED -> COMPLETED`，分支状态包括 `CANCELLED` 与 `FALLBACK_TICKET`。状态机保证了流程可控与可追踪。

## 第五章 系统详细设计与实现

### 5.1 会话与 AI 对话模块

系统支持会话创建、重命名、删除、消息历史查询。AI 对话时，先构建上下文（近期消息 + RAG 片段），再调用模型生成回答，最终持久化消息并返回前端。

### 5.2 RAG 知识增强模块

1. 文档上传后通过解析组件提取文本；
2. 文本按分块策略切分；
3. 调用 embedding 服务向量化后写入 Qdrant；
4. 查询时根据用户问题生成 query vector，在租户范围内 TopK 检索；
5. 将检索结果拼接为 prompt 上下文并返回引用来源。

该模块有效提升了领域问答准确度与可解释性。

### 5.3 转人工模块（核心）

#### 5.3.1 发起转人工

用户发起后，系统校验会话归属并检查是否重复申请。若无在线客服，则执行工单兜底策略并返回 fallback 信息；若有在线客服，则创建 handoff 请求并进入排队。

#### 5.3.2 客服分配与接入

系统异步分配客服并推送 `HANDOFF_REQUEST`；客服接受后更新状态为 `ACCEPTED`，同步更新会话模式与工单状态，并通知用户端。

#### 5.3.3 人工会话与结束

接入后双方通过 WebSocket `TEXT` 实时通信。会话可由客服完成或用户主动结束，两者最终收敛为 `COMPLETED`，并通过 `endedBy` 在事件与消息层区分发起方。

#### 5.3.4 事件审计

关键操作记录到 `handoff_event`，形成完整时间轴，便于问题追踪与运营分析。

### 5.4 客服状态管理模块

客服状态存储于 Redis，核心实现如下：

- `agent:status:{tenantId}:{userId}`：Hash 存储状态与心跳时间；
- `agent:sessions:{tenantId}:{userId}`：当前会话数；
- `agent:online:{tenantId}`：在线客服 ZSet。

前端定时心跳刷新 TTL，服务端据此判定在线有效性；分配前通过 `canAcceptSession` 检查状态与容量，防止超载。

### 5.5 工单与闭环模块

在负面情绪或转人工失败场景，系统自动创建工单并进入处理流程。通过异步事件机制实现工单创建、通知、状态流转，增强系统容错能力。

## 第六章 系统测试与结果分析

### 6.1 功能测试

围绕以下主线进行验证：

- C 端：会话管理、AI 对话、转人工申请/取消/结束；
- 客服端：请求接入、实时聊天、完成会话；
- 管理端：工单处理、状态监控；
- 多租户：跨租户访问隔离校验。

测试结果表明，核心业务流程均可闭环执行，关键状态变更与消息推送符合预期。

### 6.2 性能与稳定性观察

在中等并发下，系统整体响应稳定。RAG 查询相较纯对话耗时有所增加，但答案相关性显著提升。Redis 缓存与异步队列机制有效降低了关键路径压力。

### 6.3 问题与改进

当前系统仍可在以下方面优化：

- 向量检索参数动态调优策略可进一步细化；
- 管理员端数据看板需要更完善的统计图表；
- 长对话上下文压缩策略可进一步提升 token 利用率。

## 第七章 部署与运维设计

系统支持前后端分离部署，后端与中间件可通过 Docker Compose 编排。核心部署项包括 MySQL、Redis、Qdrant、RabbitMQ、应用服务。通过日志与监控可观测关键指标（错误率、响应时间、在线客服数、队列积压等）。

## 第八章 总结与展望

本文围绕多租户智能客服中台，完成了从需求分析、架构设计到核心模块实现与测试验证的系统性工作。系统在 AI 问答、RAG 增强、转人工协同、工单兜底与租户隔离方面实现了业务闭环，并具备较好的工程可复用性。

未来工作可聚焦于：

1. 引入混合检索（向量 + 关键词）提升召回效果；
2. 完善管理员运营看板与 SLA 监控；
3. 增强知识更新自动化与质量评估机制；
4. 支持更多接入渠道与更细粒度权限模型。
