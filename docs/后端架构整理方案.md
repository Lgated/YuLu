## 📦 三、整理方案

### 3.1 删除冗余Controller

#### 3.1.1 删除 `user.controller.AuthController`

**原因**：
- 功能已被 `AdminAuthController` 完全替代
- 路由 `/api/auth/*` 已被 `/api/admin/auth/*` 替代

**影响**：
- ✅ 无影响，前端已使用新接口

**操作**：
```bash
删除：src/main/java/com/ityfz/yulu/user/controller/AuthController.java
```

#### 3.1.2 删除 `chat.controller.ChatController`

**原因**：
- 功能已拆分：
  - C端：`CustomerChatController` (`/api/customer/chat/*`)
  - B端：`AdminSessionController` (`/api/admin/session/*`)

**影响**：
- ⚠️ 需要检查是否有前端代码仍在使用 `/api/chat/*` 接口
- 如果有，需要迁移到对应的B/C端接口

**操作**：
```bash
删除：src/main/java/com/ityfz/yulu/chat/controller/ChatController.java
```

#### 3.1.3 删除 `ticket.controller.TicketController`

**原因**：
- 功能已被 `AdminTicketController` 完全替代
- 路由 `/api/ticket/*` 已被 `/api/admin/ticket/*` 替代

**影响**：
- ⚠️ 需要检查是否有前端代码仍在使用 `/api/ticket/*` 接口

## 📝 八、总结

### 8.1 整理收益

- ✅ **职责清晰**：B/C端分离，业务模块独立
- ✅ **代码精简**：删除冗余Controller，减少维护成本
- ✅ **结构规范**：DTO按用途分类，便于管理
- ✅ **统一标准**：ErrorCodes统一管理

### 8.2 整理原则

1. **B/C端分离**：B端功能放在`admin`包，C端功能放在`customer`包
2. **业务模块独立**：`chat`、`ticket`等业务模块只保留entity、mapper、service
3. **通用组件抽离**：通用DTO放在`common.dto`，通用枚举放在`common.enums`
4. **避免重复**：删除功能重复的Controller和类
